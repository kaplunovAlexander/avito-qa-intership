# Avito QA Internship — API test automation solution

Решение покрывает задание 2.1: API-тесты для сервиса объявлений.  
В проекте есть:
- автоматизированные тесты на `pytest + requests`
- отчёты `Allure`
- конфигурация линтера и форматтера через `Ruff`

## Состав файлов

- `BUGS_SCREENSHOT.md` — баги, найденные на скриншоте с первого задания
- `TESTCASES.md` — тест-кейсы
- `BUGS.md` — дефекты, найденные в процессе прогона
- `api_client.py` — клиент для работы с API
- `data_factory.py` — генерация тестовых данных
- `allure_helpers.py` — вложения и нормализация ответов для Allure
- `conftest.py` — фикстуры и хуки pytest
- `test_api.py` — автотесты
- `requirements.txt` — зависимости
- `pytest.ini` — конфигурация pytest
- `pyproject.toml` — конфигурация Ruff
- `categories.json` — категории дефектов для Allure

## Почему выбран такой стек

`pytest` удобен для коротких, читаемых API-тестов и поддерживает фикстуры и маркировку тестов из коробки.

`Allure` официально поддерживает интеграцию с `pytest`, шаги, метаданные и вложения через адаптер `allure-pytest`. Для генерации результатов используется `--alluredir`.

`Ruff` можно использовать и как линтер, и как форматтер, с конфигурацией через `pyproject.toml`, что уменьшает число инструментов и делает проект проще.

## Требования

- Python 3.11+
- доступ в интернет до `https://qa-internship.avito.com`
- установлен Java Runtime, если вы хотите открыть Allure Report локально
- установлен Allure CLI

## Установка

### 1. Клонировать репозиторий
```bash
git clone <URL_ВАШЕГО_РЕПОЗИТОРИЯ>
cd avito-qa-internship
```

### 2. Создать виртуальное окружение
#### Windows
```bash
python -m venv .venv
.venv\Scripts\activate
```

#### macOS / Linux
```bash
python3 -m venv .venv
source .venv/bin/activate
```

### 3. Установить зависимости
```bash
pip install -r requirements.txt
```

## Запуск тестов

### Все тесты
```bash
pytest
```

### Только smoke
```bash
pytest -m smoke
```

### Только негативные
```bash
pytest -m negative
```

### Только E2E
```bash
pytest -m e2e
```

## Allure

### Сохранить результаты Allure
```bash
pytest --alluredir=allure-results --clean-alluredir
```

### Открыть отчёт локально
```bash
allure serve allure-results
```

### Что уже добавлено в тесты для Allure
- `@allure.title(...)`
- `@allure.description(...)`
- `allure.step(...)`
- вложения request/response
- текст ошибки при падении теста

## Линтер и форматтер

### Проверка кода линтером
```bash
ruff check .
```

### Автоисправление части проблем
```bash
ruff check . --fix
```

### Форматирование
```bash
ruff format .
```

## Особенности тестов

1. Стенд общий для всех кандидатов, поэтому тесты:
   - не используют фиксированный sellerId
   - создают собственные данные
   - не опираются на фиксированное количество объявлений

2. Для sellerId используется диапазон `111111–999999`, как рекомендовано в задании.

## Что считается багом и выносится в BUGS.md

Если фактическое поведение противоречит ожидаемому, например:
- невалидные данные принимаются
- статус-код не соответствует логике ошибки
- v1 и v2 возвращают противоречивую статистику
- GET по созданному id не находит только что созданную запись
- структура ответа не соответствует ожидаемому контракту

Тогда это фиксируется в `BUGS.md`.

## Примечание

Решение написано так, чтобы тест:
- либо проходит
- либо падает с понятной причиной
- не является «сломавшимся» без объяснения
